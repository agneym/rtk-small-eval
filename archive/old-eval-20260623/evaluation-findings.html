This single instance accounts for three-quarters of the resolved-only token increase. The root cause is a pytest rewrite bug:

  1. RTK rewrote python -m pytest ... → rtk pytest ...
  2. The rewritten command returned Pytest: No tests collected
  3. The agent saw the failure and tried to debug it — for 70 extra turns
  4. Late turns each cost ~36k–41k tokens (full prompt history)
This is not an outlier to dismiss. Smaller "Pytest: No tests collected" events appeared in 6 other RTK-on runs: astropy-14182 (3×), astropy-6938, django-10914, django-10924, django-11039, django-11049. The pytest rewrite problem is systematic, not a one-off.

Root cause diagnosis

Why RTK consumed more tokens than it saved

The four mechanisms

  1. Prompt-token amplification from extra calls. The +2.18M token increase is overwhelmingly prompt tokens (+99.5% of the delta), not longer assistant responses. Each extra turn repays the full conversational history.
  2. RTK changes command semantics. The worst case is pytest. python -m pytest --help behaved like rtk pytest --help, and test runs returned successful No tests collected. The agent mistrusted the environment and went into a debugging spiral. This is not a token-compaction problem — it's a behavior preservation problem.
  3. Compact outputs remove cues the model relies on. Compact find/ls summaries can be useful, but they can also force the agent to issue follow-up commands to recover exact paths or context. The savings on any single observation are dwarfed by the cost of an extra turn.
  4. Late extra calls are exponentially expensive. In astropy-14995, the final calls each cost ~36k–41k tokens because the prompt history had grown. The last 20 unnecessary-ish calls cost hundreds of thousands of tokens even though their raw outputs were often tiny.

Recommendations

  1. Do not rewrite pytest or test-runner commands by default. Leave pytest, python -m pytest, ./manage.py test, tox, and similar commands untouched unless RTK can exactly preserve exit status and semantics. Test commands are pass/fail verification signals, not bulk information retrieval.
  2. Never return success for "No tests collected" unless the underlying command would. This is especially dangerous for SWE-bench agents because verification failures guide the next step. Returning a clean exit code with a misleading message sent the agent into a debugging loop.
  3. Use RTK mainly for read-only high-volume inspection commands. Safer targets: cat on large files, broad grep, broad find, maybe ls. Riskier targets: tests, scripts, commands with meaningful exit status, commands in pipelines where downstream tools expect exact output.
  4. Add a fallback/escape rule. If an RTK-rewritten command returns suspicious output (No tests collected, parser/help text, timeout, or command-not-found), automatically rerun the original command once without RTK and show both outputs.
  5. Measure turn count as a first-class RTK metric. Token savings should be reported alongside call deltas and loop indicators. RTK is only net-positive when it reduces observation tokens without increasing the number of turns.

Caveats & methodology

Experimental design

  • 23 paired instances from SWE-bench Lite across 5 independent slices.
  • Each pair runs the same instance with the same model — once with RTK command rewriting (on) and once without (off).
  • Two models tested: MiniMax M3 (4 pairs, slice 0:5) and DeepSeek V4 Flash (19 pairs, slices 5:25).
  • Hard caps (per-instance token budget + step limit) were identical across both arms.
  • Resolution graded with the standard SWE-bench harness (SWE-bench/SWE-bench_Lite dataset).

Exclusions

  • astropy__astropy-14365 — excluded entirely; rtk-off run hung, no paired trajectory.
  • django__django-11019 — excluded entirely; incomplete paired trajectory.

What this evaluation does NOT test

  • This tests RTK as an automatic command rewriter in an agent loop (similar to RTK's auto-rewrite hook behavior). It does not test interactive RTK use by a human developer.
  • The results are specific to SWE-bench Lite — a self-contained bug-fixing benchmark. RTK may behave differently on other agent workloads.
  • Only two models were tested. Different models may respond differently to rewritten observations.

Interpretation

The takeaway is not that RTK is a bad tool. It is that transparent automatic rewriting of agent commands has second-order effects that can overwhelm first-order token savings. The pytest rewrite case demonstrates that even correct-looking rewrites (compacting test output) can break the agent's verification loop. RTK's value proposition — compacting large command output — is real, but must be scoped to commands where the risk of behavior change does not trigger extra agent turns.
